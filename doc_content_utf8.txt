4. Pool Maintenance System — Full Specification
This is the most structured module in the system and requires the most precise implementation. Every rule below is a hard operational requirement, not a suggestion. The pool module must enforce these rules automatically — staff cannot skip or bypass them.
4.1  Daily Pool Schedule — Fixed Rules
The following three rules are non-negotiable operational requirements and must be enforced by the system every single day:
R1
Pool Pump / Motor — Minimum 3 Hours Daily Runtime
The pool pump must run for a minimum of 3 continuous or accumulated hours every day. The system must track pump runtime automatically and alert staff if the daily minimum has not been reached by 6 PM.
System behaviour: 
Staff log pump ON and pump OFF times. System calculates runtime. If runtime < 3 hours by 18:00, an alert fires and a task is created to run the pump.
R2
Strainer Check — Mandatory Before Every Pump Start
The strainer basket must be cleaned and checked before the pump is switched on each time. Additionally, the strainer must be checked at regular intervals while the pump is running.
System behaviour: 
Staff cannot log pump ON until they have first submitted a strainer check entry. The pump start log form shows: Strainer cleaned? [Yes/No] — if No is selected, the form cannot be submitted and a warning is shown.
R3
Pool Bottom Vacuuming — Every 2 Days
The pool floor must be vacuumed once every two days. The system tracks the last vacuum date and automatically creates a vacuuming task on the correct days.
System behaviour: 
After each vacuum log is submitted, the system sets the next due date to +2 days. On that date, a task is auto-created at 7 AM and assigned to pool staff. If not completed by noon, it escalates to the manager.

8. WhatsApp Notification — Simulation Specification
Because WhatsApp Business API requires Meta business verification (2–4 weeks, plus approval is not guaranteed for new accounts), the demo version must include a built-in simulation that is indistinguishable from the real experience. This simulation is not a placeholder — it is a complete feature that will also be useful in production for testing and monitoring.
⚠  
Do not display any disclaimer or 'simulation mode' label visible to the end user during a demo walkthrough. The simulation panel should look and feel exactly like real WhatsApp messages. The 'simulation' label should only appear in developer/admin settings.
8.1  The Simulation Panel — What to Build
Build a dedicated panel inside the app called the Notification Centre. This panel has two views:
Manager View — Notification Centre
▸ 
A feed of all outgoing notifications sent from the system, shown in WhatsApp-style message bubbles
▸ 
Each message shows: recipient name, message content, timestamp, and delivery status (Sent / Delivered / Read / Acknowledged)
▸ 
Filter by staff member, notification type, date range, or status
▸ 
Unacknowledged alerts highlighted in amber — manager can manually follow up
▸ 
A Send Now button to manually push any notification immediately
▸ 
Notification log is exportable as a PDF or CSV report
Staff View — Notification Inbox
▸ 
A dedicated screen on the staff interface that looks exactly like a WhatsApp chat thread
▸ 
Messages arrive here in real time using Socket.IO — no page refresh needed
▸ 
Each message has a Confirm / Help Needed reply button below it
▸ 
Confirming a message marks the task as Acknowledged in the system — same as a WhatsApp reply would
▸ 
Unread messages show a red badge count on the inbox icon
▸ 
Morning briefing arrives at 7 AM as a structured list of the day's tasks
8.2  Message Format — Exactly As It Will Appear
Every notification must follow this exact format so that when real WhatsApp is connected, the messages are identical. Developers should build the message templates as reusable components.

8.4  Switching to Real WhatsApp in Production
When the WhatsApp Business API is approved and integrated, the only change required should be replacing the in-app delivery method with the API call. All message templates, triggers, reply handling, and escalation logic remain identical. Build the simulation with this in mind — it should be a drop-in replacement, not a rewrite.
What changes when real WhatsApp is connected
▸ 
Messages are sent to staff phone numbers via the WhatsApp Business API instead of the in-app inbox
▸ 
Staff reply via their actual WhatsApp on their personal phone — the webhook receives the reply and processes it identically to a simulation reply button click
▸ 
The Notification Centre continues to display all messages and statuses, now as a monitoring layer
▸ 
No changes to trigger rules, message templates, escalation logic, or database structure
